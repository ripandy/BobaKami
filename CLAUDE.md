# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

BobaKami is a Unity 6 (`6000.3.1f1`) face-detection game. (The name is a pun:
*boba* = tapioca, *kami* = god/bite.)
The player uses ARKit face tracking to turn their head (aim direction) and bite
(mouth blendshape) to eat falling boba beans before too many drop. Targets iOS/ARKit;
also has WebGL and macOS build configs.

## Project setup

This project depends on **sibling repositories outside this folder**. `Packages/manifest.json`
references local packages via relative paths:
- `com.ripandy.*` modules → `../../ModuleCollections/*`
- `com.ripandy.soar` (SOAR) → `../../SOAR`

The project will not resolve packages or open in Unity unless those folders exist
next to the repo (i.e. `<parent>/ModuleCollections/` and `<parent>/SOAR/`).

`.csproj` and `.sln` files are git-ignored and regenerated by Unity/Rider — never
edit them by hand. Edit code under `Assets/` and let Unity regenerate project files.

## Build & test

Open in Unity Editor `6000.3.1f1`. Build scenes (in order): `Core` → `Title` → `Gameplay`.

Tests use Unity Test Framework (NUnit). The `BobaKami.Tests` assembly is **EditMode-only**
(`Assets/1_BobaKami/Tests/`) and tests the pure-C# domain with no Unity runtime needed.

- In-editor: run via **Window → General → Test Runner → EditMode**.
- CLI (headless):
  ```
  Unity -batchmode -runTests -projectPath . -testPlatform EditMode -testResults results.xml -quit
  ```
- Run a single test/class: add `-testFilter "BobaKami.Tests.<ClassOrMethod>"`.

## Architecture

The codebase is **ports-and-adapters**: the game's rules live in a pure-C# assembly
that knows nothing about Unity, and Unity-specific code adapts to it.

### `Assets/1_BobaKami` — domain (`BobaKami` assembly)
Pure C#. The asmdef sets `noEngineReferences: true` and `autoReferenced: false`, so
**this layer must never reference UnityEngine.**
- `Entities/` — `Player`, `Bean`, `BeanLauncher`, and the `GameStateEnum` / `AppStateEnum`.
- `GameStates/` — `IGameState` (`ValueTask<GameStateEnum> Running(ct)`) and the
  concrete states (`IntroGameState`, `PlayGameState`, `GameOverGameState`). A state
  runs its logic and **returns the next state**. `PlayGameState` drives gameplay via
  async recursive loops cancelled through a `CancellationTokenSource`.
- `Interfaces/` — the **ports**: presenter interfaces (e.g. `IPlayerHealthPresenter`,
  `IBeanPresenter`) the domain pushes output to, and input-provider interfaces
  (e.g. `IPlayerDirectionInputProvider`, `IPlayerBiteInputProvider`) it awaits input from.
- `DataTransferObjects/`, `Tests/`.

### `Assets/2_Contents` — Unity adapters (`BobaKami.Contents` assembly)
Implements the domain ports using **SOAR** (a reactive ScriptableObject-architecture
framework). The recurring pattern: a SOAR `Variable<T>` / `GameEvent<T>` also
implements a domain interface, so the domain talks to a ScriptableObject without
knowing it. Examples:
- `HealthPercentageVariable : Variable<float>, IPlayerHealthPresenter`
- `BittenBeanGameEvent : GameEvent<int>, IPlayerBiteInputProvider`
- `PlayerData : JsonableVariable<Player>` (serialized domain state as a SO asset)

MonoBehaviours render by subscribing reactively, e.g. `HealthBarHUD` calls
`healthPercentageVariable.Subscribe(...)`. Folders are feature-grouped under
`Gameplay/` (`Player`, `Boba`, `HUD`, `InputProviders`, `LevelIntro`, `GameOverDialog`).

### Dependency injection — Doinject
- `GameplayBindingInstaller : MonoBehaviour, IBindingInstaller` wires everything in
  `Install(DIContainer, ...)`: it binds the domain game-states as singletons and
  binds each SerializeField ScriptableObject as the domain interface it implements
  (presenters + input providers).
- `GameplayStateMachine : MonoBehaviour, IInjectableComponent` receives the states
  via `[Inject] Construct(...)` and runs the loop: repeatedly `await state.Running(ct)`,
  switch to the returned `GameStateEnum`, and on `None` execute the SOAR `resetAppCommand`.

### App flow & networking
- Scene/app-level state is handled by the external `com.ripandy.appstatemanagement`
  module (`AppStateEnum`: Splash → MainMenu → Gameplay) across the `Core`/`Title`/`Gameplay` scenes.
- `Assets/4_Network` + `NetworkManagement` / `BlendShapeBroadcast*` handle MagicOnion
  client–server streaming of face blendshape data.

## Conventions
- Keep all game logic in `BobaKami` (no Unity types). To surface new state/input,
  add an interface in `Assets/1_BobaKami/Interfaces`, implement it as a SOAR
  Variable/GameEvent in `Assets/2_Contents`, then bind it in `GameplayBindingInstaller`.
- Reactive stack: R3, UniTask, LitMotion (tweening). Prefer UniTask/`ValueTask`
  for async over coroutines in new code, matching existing files.
